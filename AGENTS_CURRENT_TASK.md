## Important Information

- Added `SymbolicResolverDefinition` / `SymbolicResolverDescriptor`. Resolver nodes are currently literal, local scalar field, relative symbol field path, global symbol field path, type size, and binary arithmetic operation. `LocalField` resolves a bare scalar already collected in the current struct instance; `RelativeSymbolField` resolves a structured path rooted at the current symbol instance; `GlobalSymbolField` resolves a module-rooted structured path such as `game.exe` + `Globals.item_count`. Binary operations are intentionally fixed to two children for simpler editing and operator swapping.
- `ProjectSymbolCatalog` now stores reusable symbolic resolver descriptors.
- `SymbolicFieldCountResolution` supports `Inferred` and `Resolver(String)`. `SymbolicFieldOffsetResolution` supports `Sequential`, `Static(u64)`, and `Resolver(String)`. Dynamic field layout no longer supports inline expression ASTs; the old `symbolic_expression` module and API re-export were removed. The string boundary format for dynamic layout is `resolver(id)`, and static offsets accept decimal, hex, or binary numeric text.
- The current resolver evaluator supports local struct-instance formulas, relative symbol chains, module-rooted global symbol chains for scalar reads, and pointer-chain resolver callbacks. Symbol Explorer now backs both relative and global pointer-chain resolver nodes with its existing pointer virtual snapshot lane. Relative pointer chains resolve symbolic links against the current struct layout before being rooted at the current symbol instance locator. Conditional type/layout selection remains deferred.
- The current runtime resolver evaluates formulas/resolvers against scalar fields collected from the same struct instance. Symbol Explorer feeds those scalar values through a dedicated virtual snapshot lane so formulaic layouts can converge over frames.
- Local field-name dependency cycles are now validated through `ProjectSymbolCatalog` using resolver descriptors that read local fields. Struct and resolver saves reject cycles such as field A's layout depending on field B while field B's layout depends on field A. Runtime resolver-stack detection still catches deeper relative/global traversal cycles that cannot be proven from the local struct graph alone.
