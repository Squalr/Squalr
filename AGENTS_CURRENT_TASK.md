## Important Information
Append important discoveries. Compact regularly ( > ~40 lines, compact to 20 lines)

- Pointer scanning uses dedicated API/session types: `PointerScanSession`, `PointerScanNode`, `PointerScanLevel`, and `PointerScanSummary`. The active session is stored in `EnginePrivilegedState`, and the top-level command surface is `pointer-scan reset|start|summary|expand|validate`.
- `PointerScanExecutor` now walks snapshot memory with native-width aligned loads, classifies matches into per-level `static` and `heap` candidate sets, and advances the frontier through heaps only. `PointerScanSession` stores compact level candidates plus summary counts, and exact display rows are materialized lazily during `pointer-scan expand` instead of eagerly building a global child-graph for every discovered pointer.
- The GUI pointer scanner now has a real window composed of a toolbar plus lazy results. The toolbar owns target address, pointer size, max depth, offset, a real `New` session reset action, a primary `Start/Validate` action, refresh, copy/export, add-to-project, and status text.
- The pointer-scanner toolbar now uses themed `DataValueBox` inputs for target / validation / depth / offset, a restricted `DataTypeSelectorView` for `u32` / `u64`, and themed action buttons. The results grid now uses column separators and mono text for address-heavy fields.
- Pointer-scan collection and validation now build merged interval frontiers via `PointerScanTargetRangeSet` instead of probing raw target-address bags. The scan path chooses a frontier kernel per level (`SIMD Linear`, `Scalar Linear`, or `Scalar Binary`) and collection fans that kernel across snapshot regions in parallel.
- The old `PointerScanTargetRange` helper is gone. `PointerScanTargetRangeSet` now stores `NormalizedRegion` directly and maps the scanner's inclusive `[lower, upper]` reachability checks through the existing `base_address` / `end_address` API instead of carrying a redundant wrapper type.
- Project-explorer address items now expose a `Pointer Scan` context-menu action that opens the pointer-scanner window and autofills the target inputs. Module-relative address items currently warn that the stored offset is not auto-resolved to a live absolute address before scan start.
- Pointer project items now persist a real chain: root `address`, `module`, `pointer_offsets` as `Vec<i64>`, `pointer_size`, symbolic struct reference, and preview display value. Pointer project item creation is supported through the unprivileged create command and writes a real `.json` project item under the hidden project root.
- Project item preview refresh now resolves stored pointer chains at runtime before reading values. Freeze activation, freeze request execution, and the periodic freeze task also resolve full pointer chains before reading or writing memory.
- Pointer project-item list refresh now also records a transient evaluated pointer path string from the same live chain walk used for preview reads, and the project-explorer tree renders that path inline beside pointer names while keeping the preview value right-aligned.
- Pointer scans launched from a typed project address now carry that symbolic data type through `PointerScannerViewData`, so created pointer project items no longer hardcode `u8` preview reads; project-explorer pointer rows also now use the value-type icon and show the full evaluated chain as a hover tooltip when the inline path text is truncated.
- The shared struct viewer now has a small field-presentation layer. Project-item properties that persist `symbolic_struct_definition_reference` are rendered as a friendly `data_type` row with `DataTypeSelectorView`, so both address and pointer property viewers can edit the stored symbolic data type without treating it like a raw string field.
- Focused pointer-preview tests now prove `ProjectItemsListRequestExecutor` resolves the pointer chain first, then issues a separate final memory read at that resolved address, including negative offsets and after project-item serde round trips. - Pointer-scanner reset now invalidates in-flight summary/start/validate revisions before late callbacks can restore a cleared session, clears the local tree immediately on `New`, and keeps `New` / `Start|Validate` / refresh actions disabled while reset is still in flight. - Pointer-scan collection now preserves natural snapshot/task traversal order instead of reducing worker-local vectors into an unordered merged level and sorting afterward. Heap candidates from collection are now flattened in task order, fed directly into `PointerScanTargetRangeSet::from_sorted_target_addresses_iter(...)`, and persisted as presorted heaps so the next frontier no longer pays an extra sort just to recover address order already present in the scan traversal.
- The initial scan top-level now mirrors the old C# shape more closely too. `PointerScanExecutor` is back to being a thin stage runner (`collect values -> discover levels -> build session`), while reusable helpers now own the lower-level work in `pointer_scan_level_collector.rs` and `pointer_scan_session_builder.rs`.
- Pointer-scan validation now mirrors the old C# `PointerRetarget` / `PointerRebase` mental model much more closely: start from the validation target as the first frontier, rebuild each heap level from the live heap snapshot with the same range-search kernels used by collection, then reread only the stored static bases for that level and keep the ones whose live values still reach the frontier.
- The validator no longer replays stored heap graphs or rebases child addresses one parent at a time. Each level is now just `frontier -> live heap scan -> stored static prune -> next frontier`, which makes the code easier to reason about and also allows validation to rediscover newly live heap candidates from the current snapshot rather than being limited to the old retained heap set.
- Validation heap scans now carve heap-only scan tasks out of the fresh snapshot by subtracting module ranges up front, so heap rebuilding no longer pays per-match static/heap classification during validation and the level pipeline stays structurally close to collection.
- Pointer-scan execution logging now reports target metadata up front, per-level frontier/range/kernel summaries, per-level elapsed time, final retained-node counts, and total collection / validation time. Validation logging still reports periodic heap-region progress so long validations no longer go silent after value collection.
- Root/session finalization was still a hotspot after scan completion. The current heap-frontier design means every retained static candidate is already a root, so collection/validation now set `root_node_count = total_static_node_count`, and `PointerScanSession::materialize_root_node_page` now pages directly over retained statics instead of cloning intermediate root candidate buffers.
- Pointer-scan collection and validation were still retaining heap candidates on the terminal discovery depth even though those heaps cannot become roots or feed any deeper static level. The terminal level now keeps only static candidates, and focused coverage asserts that terminal heap nodes are omitted.
- Profiling later showed three remaining hot scalar paths: `contains_value_binary`, `find_matching_range_index_binary`, and pointer-value decoding during scan / validation. `PointerScanTargetRangeSet` now stores flattened bounds plus coarse 64KB bucket spans so large binary searches only probe the relevant range slice, while pointer scan kernels decode scalar/SIMD lane values through direct unaligned loads instead of per-element slice conversions.
- Validation static pruning now just rereads stored static bases by module rebasing and checks those live values against the current frontier range set. The old ordered child reread path and heap append cleanup are gone with the replay design.
- Pointer-scan collection now builds its Rayon scan tasks directly from snapshots and sizes large-region splits adaptively from total snapshot bytes and available workers, instead of always flattening snapshot regions first and forcing every large region into fixed 1MB tasks.
- Pointer-scan kernels were still materializing a temporary `Vec<PointerScanRegionMatch>` and then immediately copying those matches into candidate vectors. Collection and validation now stream matches directly into their per-task candidate buffers, and frontier range-set creation now uses `from_sorted_target_addresses(...)` once the caller has already sorted/deduplicated the frontier instead of cloning and sorting it again.
- Pointer-scan collection and validation were still allocating one candidate buffer per Rayon task after the 1MB chunking pass. Those paths now use Rayon `fold(...)` so each worker thread reuses the same candidate vectors across many tasks, and validation also reuses one 64KB scan buffer per worker instead of one per region.
- Internal pointer-scan helper structs in `squalr-engine-scanning` now live under `squalr-engine-scanning/src/pointer_scans/structures/` with one struct per file where it made sense, keeping executor/kernel/validator files focused on behavior instead of nested type declarations.

- Pointer-scan search code no longer lives in one monolithic `pointer_scan_range_search_kernel.rs`. The logic now sits under `squalr-engine-scanning/src/pointer_scans/search_kernels/`, with a small facade plus separate kernel-kind, scalar, SIMD, and pointer-read helpers.
- Pointer-scan discovery/validation no longer bounce frontier targets through per-level sorted `Vec<u64>` address buffers. They now carry `PointerScanTargetRangeSet` forward directly between levels, build new frontiers from sorted candidate iterators, and validation also borrows static-candidate slices instead of cloning whole level vectors just to read them.

Static discoveries now carry `module_index + module_offset` until session finalization, frontier rebuilding sorts only raw pointer addresses where ordering is actually required, and the persisted pointer-scan session now stores one shared module-name table plus `module_index` static candidates instead of cloning the same module string into every retained static node.
