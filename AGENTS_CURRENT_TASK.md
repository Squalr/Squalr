That means the current header-based heuristic can detect the presence/count of linked GBA blocks but does not differentiate slot identity by itself; any `gba_wm_<slot>` / `gba_im_<slot>` assignment needs an additional ordering or slot-metadata heuristic beyond the cartridge header.
- Dolphin source confirms that integrated mGBA slot identity is the SI channel / `device_number` end-to-end: `CSIDevice_GBAEmu` constructs `HW::GBA::Core(system, m_device_number)`, `CoreInfo.device_number` is surfaced to the host/UI/config layer, and Qt widget/title/config plumbing keeps using that same index. That means slot identity definitely exists inside Dolphin, but it lives in C++ object state rather than in the generic GBA ROM header; from raw external memory alone we should expect reliable `0x48000` block detection before reliable slot attribution unless we find a correlated runtime table in `gc_wii` or a stable ordering invariant in Dolphin's host allocations.
- mGBA source confirms the observed `0x48000` candidate shape is just one contiguous anonymous allocation for `EWRAM (0x40000) + IWRAM (0x8000)`: `GBAMemoryInit` allocates `anonymousMemoryMap(GBA_SIZE_EWRAM + GBA_SIZE_IWRAM)` and then points `iwram` at `&wram[GBA_SIZE_EWRAM >> 2]`. The allocation itself carries no slot metadata. Joybus/link handling and any per-device identity live in the surrounding `GBA` / `GBASIO` / Dolphin `CSIDevice_GBAEmu` objects and in I/O registers, not inside that raw `0x48000` RAM block. So the block does not self-identify generically in mGBA.
