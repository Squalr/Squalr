- The Struct Viewer external-viewer button policy used to be split awkwardly: view-data said `MemoryViewerButton`, then the row widget re-inspected the validation data type and sometimes silently turned that into a code-viewer action. This pass makes the presentation explicit with a real `CodeViewerButton` variant so the enum matches behavior, and the x86/x64 instruction-array case is now decided once in `StructViewerViewData` instead of ad hoc inside `StructViewerEntryView`. Reverified with `cargo fmt --all`, `cargo test -p squalr struct_viewer -- --nocapture`, and `cargo check -p squalr`.
- Instruction-family plugin default enablement is now a target-arch policy decision rather than a hardcoded global-on choice. Each built-in instruction plugin still registers everywhere, but its `PluginMetadata.is_enabled_by_default` flag now follows the compile target family so the default plugin set is less noisy on each platform while preserving manual opt-in for cross-ISA work. Reverified with `cargo fmt --all`, `cargo test -p squalr-plugin-instructions-x86`, `cargo test -p squalr-plugin-instructions-arm`, `cargo test -p squalr-plugin-instructions-powerpc`, and `cargo check -p squalr`.
- The x86 plugin was duplicating the engine’s `Bitness` concept under the misleading local name `X86InstructionMode`, even though it only represented 32-bit vs 64-bit width and had no x86-specific mode semantics. This pass removes that duplicate enum and threads shared `Bitness` through the x86 instruction-set/memory-operand/candidate-lowering code, keeping only tiny local conversions where `iced_x86` requires numeric bitness values. Reverified with `cargo fmt --all`, `cargo test -p squalr-plugin-instructions-x86`, and `cargo check -p squalr`.
- The old `DataTypeI*` names were leaking shorthand transport IDs into the Rust API surface, which made the type names read like abbreviations instead of concrete instruction-family data types. This pass keeps the short `i_*` string IDs exactly as before for registry/plugin compatibility, but renames the Rust structs/modules to explicit semantic names and updates the x86 helper filename to describe its actual job. Reverified with `cargo fmt --all`, `cargo test -p squalr-plugin-instructions-x86`, `cargo test -p squalr-plugin-instructions-arm`, `cargo test -p squalr-plugin-instructions-powerpc`, and `cargo check -p squalr`.
- Symbol-store planning is now captured in `docs/symbol_store.md` as an execution plan instead of a broad deep dive. The low-regret direction is: add named layout fields first, reuse pointer-scan pointer encodings in symbolic containers, add rooted project-owned symbols for `absolute` and `module + offset`, keep module names as strings for now, keep address/pointer items as promotion sources, derive children lazily, and keep external-tool quirks in metadata instead of hardcoding Ghidra/IDA-shaped core types.
- The symbol-store plan now also calls out the managed-runtime extension seam explicitly: virtual modules and data types remain the base, while CLR/JVM-style quirks should land later through a runtime/symbol-provider plugin that can contribute managed roots, resolution logic, and runtime-specific metadata without reshaping the core symbol model.
- The symbol-store plan now also settles the intended GUI shape: a dedicated Symbol Explorer should be the main symbol-authoring surface and should behave more like Project Explorer than Memory Viewer or Code Viewer. Memory/Code viewers stay page-oriented consumers of symbol labels with jump/promote actions, while Project Explorer keeps raw address/pointer acquisition items separate from the authored symbol tree.
- The symbol-store plan now states the project-model boundary explicitly: rooted symbols are their own stored symbolic records, not just decorated address/pointer items. Address and pointer items remain acquisition/workflow tools, and `Promote to Symbol` is the bridge that creates a real rooted symbol in the project symbol store.
- The symbol-store plan now gives the TUI an explicit track too: initial TUI symbol support should be tree-first in the project workspace, closer to the existing TUI project explorer than to page-oriented memory/code views. The current TUI memory viewer path is still relatively narrow and address-centric, so some TUI memory/code cleanup may be needed before deeper symbol-driven viewer integration feels good.
- The symbol-store plan now gives CLI explicit coverage too: symbol support should not be treated as "automatically covered" just because GUI/TUI use commands. When symbol workflows stay unprivileged-side, we should add dedicated unprivileged command paths so CLI can list/create/promote/inspect/rename/delete symbols without depending on GUI-only orchestration.
- The current ownership split is directionally correct for symbol authoring: projects persist symbols unprivileged-side, project open/close syncs them into the privileged symbol registry for execution-time reads/writes, and the unprivileged side also keeps local-project resolution for struct layouts. The main scalability hazard is that privileged registry refresh is still whole-catalog by generation, so large authored symbol stores would be expensive over Android or other IPC-heavy deployments.
- Virtual modules already have the right extension seam through `MemoryViewInstance` (`get_modules`, `address_to_module`, `resolve_module_address`, pointer-scan region overrides), and the Dolphin plugin proves we can label guest memory as module-like roots. A future symbol store should reference stable module ids from that layer instead of inventing a second virtual-module abstraction.
- Weird architecture support is inconsistent today: pointer scans already model `u24` / `u24be` / `u32` / `u32be` / `u64` / `u64be`, but symbolic field/container modeling still hardcodes only `Pointer32` and `Pointer64`. Unifying pointer/container encoding is a prerequisite before a symbol store can safely support unusual targets.
- The best low-regret implementation order looks like: generalize pointer/container encoding, add rooted symbol instances beside reusable type/layout definitions, keep the full authored symbol store unprivileged-owned, send only compact execution-oriented symbol data to the privileged side, and build scan/address labeling from exact-address and inline-field matches before attempting any full reverse pointer-map style symbol resolution.
